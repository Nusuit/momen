# Momen Architecture

This project is now organized so the root `lib/` directory stays thin. App composition, routing, shared infrastructure, and feature code each have their own home.

## Top-Level Folders

- `android/`: Native Android runner generated by Flutter. This is where Android package id, permissions, Gradle config, Firebase `google-services.json`, and release signing will live.
- `ios/`: Native iOS runner generated by Flutter. This is where bundle id, permissions, Firebase `GoogleService-Info.plist`, signing, and iOS capabilities will live.
- `web/`: Flutter web runner. Keep only web-specific app shell assets and manifest here.
- `lib/`: Dart/Flutter application source.
- `test/`: Unit and widget tests.
- `scripts/`: Local quality workflow scripts for format, analyze, tests, and coverage gate.
- `.claude/`: Local agent/task helper instructions.

## Source Folders

- `lib/main.dart`: Entry point only. It initializes bootstrap services and runs `MomenApp`.
- `lib/app.dart`: Compatibility export for `MomenApp`. It should stay tiny.
- `lib/app/`: App composition layer.
  - `bootstrap/`: Startup wiring for Supabase and Crashlytics guards.
  - `routing/`: GoRouter routes, route names, and tab route mapping.
  - `shell/`: Cross-feature app shell such as bottom navigation.
  - `momen_app.dart`: ProviderScope and MaterialApp.router setup.
- `lib/core/`: Shared infrastructure that features can use.
  - `components/`: Shared reusable UI widgets.
  - `config/`: Compile-time environment flags and keys from `--dart-define`.
  - `constants/`: Theme, spacing, icons, and style constants.
  - `errors/`: Shared Failure types and error contracts.
  - `models/`: Shared app-wide models used by more than one feature.
  - `observability/`: Crash reporting setup.
  - `persistence/`: Isar local database service.
  - `providers/`: Shared Riverpod providers for core services.
  - `services/`: SDK integration services such as Supabase startup.
  - `utils/`: Pure helper logic such as spending parsing.
- `lib/features/`: Feature-first product modules.
  - `auth/`: Onboarding, sign-in, sign-up, profile, and future auth domain/data.
  - `feed/`: Anonymous feed, camera, post detail, and future photo upload/guessing.
  - `spending/`: Caption parser, spending repository slice, dashboard, and future leaderboard.
  - `recap/`: Memories and future collaborative recap rooms.

## Dependency Direction

- `main.dart` depends on `app/`.
- `app/` may compose feature pages and core services.
- `features/` may depend on `core/`.
- `core/` must not depend on `features/`.
- Feature modules should not import each other directly.

## Runtime Configuration

Supabase and Crashlytics are guarded so local tests run without cloud credentials.

Use these defines when real credentials are available:

```powershell
flutter run `
  --dart-define=SUPABASE_URL=https://your-project.supabase.co `
  --dart-define=SUPABASE_ANON_KEY=your-anon-key `
  --dart-define=ENABLE_FIREBASE_CRASHLYTICS=true
```

Crashlytics also requires Firebase platform files generated by FlutterFire before it can be enabled safely.
